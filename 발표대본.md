제가 만든 프로젝트는 **CineVault**, "나만의 인생 영화 저장소"입니다.
영화를 **찾고, 보고, 기록하고, 내 취향까지 분석**해주는 영화 아카이브 웹 서비스입니다.

지금부터 왜 만들었는지, 어떤 기능이 있는지, 만들면서 무엇이 어려웠는지 순서로 말씀드리겠습니다. [클릭]

## 슬라이드 2 — 프로젝트 한눈에 보기 (25초)

먼저 숫자로 요약하면 — **5개 페이지**, **6가지 탐색 방식**, 그리고 **백엔드 서버 0개**입니다.

TMDB라는 영화 데이터 공개 API를 사용했고, 사용자의 기록은 Redux와 localStorage를 동기화해서 브라우저에 저장합니다. 그래서 서버 없이도 새로고침하거나 브라우저를 껐다 켜도 내 기록이 그대로 유지됩니다.

오른쪽이 실제 서비스 첫 화면입니다. [클릭]

## 슬라이드 3 — 문제 정의 (35초)

이 프로젝트는 제가 영화를 볼 때마다 느꼈던 불편함에서 시작했습니다.

영화 한 편을 보려고 하면 — 상영작은 예매 앱에서, 평점이랑 줄거리는 포털에서, 예고편은 유튜브에서 따로따로 찾아야 합니다. **정보가 다 흩어져 있는 거죠.**

그리고 막상 본 영화는 "이거 내가 봤었나? 몇 점 줬더라?" — **기록이 기억에만 의존**합니다.

또 하나, 해외 영화 사이트들은 **한국에서 개봉했는지, 국내 어떤 OTT에서 볼 수 있는지**를 안 알려줍니다.

마지막으로, 내가 어떤 장르를 좋아하는지 **객관적인 데이터가 없다**는 점. 이 네 가지가 출발점이었습니다. [클릭]

## 슬라이드 4 — 해결 방향 (25초)

그래서 각 불편함을 기능 하나씩으로 대응시켰습니다.

- 흩어진 정보는 → TMDB API로 **한 화면에 통합**하고,
- 기록할 곳이 없으니 → **별점과 한 줄 평 아카이빙**을 만들고,
- 해외 기준 정보는 → **한국 개봉작 필터와 국내 OTT·극장 안내**로,
- 취향을 모르니 → **장르·별점 분포 자동 분석**으로 해결했습니다.

이 매핑이 그대로 서비스의 5개 화면이 됐습니다. [클릭]

## 슬라이드 5 — 기술 스택 & 데이터 흐름 (35초)

기술 스택입니다. **React 18 + Vite** 기반이고, 전역 상태는 **Redux Toolkit**으로 archive와 wishlist 두 개의 슬라이스를 관리합니다. 라우팅은 React Router v6, API 호출은 Axios, 스타일은 CSS Modules를 썼습니다.

오른쪽 데이터 흐름을 보시면 — 사용자가 화면을 조작하면 React UI가 Axios를 통해 TMDB API를 호출하는 단방향 구조고요, 사용자의 기록만 Redux와 localStorage가 양방향으로 동기화됩니다.

특히 데이터 페칭은 **useReducer 기반의 커스텀 훅 useMovies**로 캡슐화해서, 화면 컴포넌트는 UI 선언에만 집중하도록 설계했습니다. [클릭]

## 슬라이드 6 — 6가지 탐색 (25초)

아래로 스크롤하면 "더 불러오기" 버튼으로 목록이 계속 누적되는 페이지네이션 방식입니다. [클릭]

## 슬라이드 7 — 검색 + 필터 (30초)

검색과 필터입니다.

검색은 타이핑할 때마다 API를 부르지 않고, **Enter나 검색 버튼을 눌렀을 때만 호출**됩니다. 입력값과 제출값 상태를 분리해서 불필요한 요청을 차단했습니다.

필터 패널에서는 **정렬, 장르 복수 선택, 최소 평점, 개봉 연도 범위**를 조합할 수 있고, 활성화된 필터 개수가 버튼에 뱃지로 표시됩니다.

그리고 핵심 기능인 **"한국 개봉작만" 토글**이 기본으로 켜져 있어서, 국내 극장에 걸렸던 영화만 보입니다. [클릭]

## 슬라이드 10 — 예고편 + 국내 감상 안내 (30초)

왼쪽, 예고편은 원래 유튜브 링크로 빠져나가는 방식이었는데, **iframe으로 앱 안에서 바로 재생**되도록 바꿨습니다. 예고편이 여러 개면 아래 썸네일로 전환할 수 있습니다.

## 슬라이드 11 — 아카이브 & 찜 목록 (25초)

기록 기능입니다.

오른쪽 찜 목록은 **볼 영화**입니다. 어느 화면에서든 하트 버튼 한 번으로 찜할 수 있고, 헤더에 찜 개수가 실시간 뱃지로 표시됩니다. [클릭]

## 슬라이드 12 — 취향 분석 (25초)

이렇게 쌓인 기록이 자동으로 분석됩니다.

총 보관 편수, 평균 별점, 최애 장르 — 세 가지 통계 카드가 위에 있고, 아래에 **장르 분포와 별점 분포 바 차트**가 그려집니다.

차트 라이브러리를 쓰지 않고 **순수 CSS로 직접 구현**했습니다. 데이터가 쌓일수록 "나는 액션을 좋아하고 평균 4.5점을 주는 사람"이라는 걸 객관적으로 확인할 수 있습니다. [클릭]

## 슬라이드 13 — 어려웠던 점 (50초)

구현하면서 어려웠던 점 세 가지입니다.

**첫째, 한국 개봉작 필터링.** region=KR만 붙이면 될 줄 알았는데, popular나 top_rated 엔드포인트는 region을 줘도 전 세계 순위가 그대로 내려와서 국내 미개봉작이 섞였습니다. 엔드포인트마다 동작이 다르다는 걸 실험으로 확인하고, 해당 탭은 discover API에 with_release_type 2|3 조건으로 우회했습니다.

**둘째, 탭·필터·검색의 상태 조합.** 6개 탭과 필터, 검색이 얽히면 어떤 API를 불러야 할지 충돌합니다. 그래서 "검색어가 있으면 search 호출 후 클라이언트 필터링, 필터가 켜져 있으면 discover로 통합, 둘 다 아니면 탭별 전용 엔드포인트" — 이렇게 **3단 분기 규칙**을 먼저 정하고 구현했습니다.

**셋째, 사용성.** 기능이 동작하는 것과 쾌적한 것은 다른 문제였습니다. 검색 호출 최적화, 페이지네이션 누적, 예고편 임베드 같은 디테일을 다듬는 데 시간을 많이 썼습니다. [클릭]

# 예상 질문 & 답변 (Q&A 대비)

**Q1. 왜 Redux를 썼나요? useState나 Context로도 되지 않나요?**
> 보관함과 찜 목록은 홈·상세·아카이브·헤더 등 여러 화면에서 동시에 읽고 쓰는 전역 상태입니다. Context로도 가능하지만, 상태 변경 로직을 슬라이스로 모아 관리하고 localStorage 동기화 지점을 한곳으로 통제하기 위해 Redux Toolkit을 선택했습니다.

**Q2. 검색에 디바운스 대신 제출 방식을 쓴 이유는?**
> 처음엔 500ms 디바운스로 실시간 검색을 구현했는데, 한글 특성상 타이핑 중간 글자로도 API가 호출돼 무의미한 요청이 많았습니다. 영화 검색은 사용자가 제목을 알고 입력하는 경우가 대부분이라, Enter 제출 방식이 요청 수와 UX 모두 나았습니다. (디바운스 훅은 직접 구현해봤기 때문에 두 방식의 트레이드오프를 비교할 수 있었습니다.)

**Q3. reducer 안에서 localStorage를 호출하면 순수 함수 원칙에 어긋나지 않나요?**
> 맞습니다. 원칙적으로는 미들웨어나 리스너에 두는 게 정석입니다. 다만 이 프로젝트 규모에서는 미들웨어가 과한 구조라고 판단했고, 사이드 이펙트 지점을 localStorage 유틸 함수로 모아 슬라이스에서만 호출하는 절충안을 선택했습니다. 규모가 커지면 redux-persist나 리스너 미들웨어로 옮길 계획입니다.

**Q4. API 키가 프론트엔드에 노출되지 않나요?**
> 네, 클라이언트 사이드 앱이라 빌드 결과물에 키가 포함됩니다. TMDB는 무료 공개 API라 위험이 낮지만, 실서비스라면 프록시 서버를 두고 키를 서버에 숨기는 구조로 가야 한다는 점을 인지하고 있습니다.
